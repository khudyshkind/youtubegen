# WORKFLOW — Правила совместной работы (Архитектор + CC)

Правила подтверждены прецедентами из реальных сессий. Каждое правило содержит обоснование и прецедент.

---

## ДЕПЛОЙ И ПРИЁМКА

### 1. Приёмка деплоя — только фактом (SHA + Ready/Online показаны)

Три уровня, все разные — все нужно проверять:
- `git log --oneline -1` — локальный коммит (не значит, что запушен)
- `git log origin/main --oneline -1` — запушен в GitHub (не значит, что задеплоен)
- Vercel UI или `vercel ls` / Railway GraphQL — развёрнут ли на проде

«Задеплоится / возьмёт / автоматически» — не приёмка. Только Ready/Online + верхний SHA показаны.

**Прецеденты**: «задеплоится автоматически» ~10 раз за июль; приёмки этапа 3 подписки шли поверх упавшего video-server (4ч24м даун); GitHub outage съел push — «SHA запушен» при коммите только локально.

---

### 2. Push в `video-server/**` = расстрел активного рендера

Рестартует контейнер Railway → убивает все processing-job'ы.

Перед пушем в video-server:
1. Проверить `video_jobs` на статус `processing`.
2. Если есть активные — подождать или предупредить владельца.

Пуши в веб-часть (`src/`, `supabase/`, корень) безопасны — Watch Paths → SKIPPED.

**Прецедент**: 17.07 — три рендера «расстреляны» пушами включая наш же пуш косметики. Это была корневая причина «видео зависает на 5%», а не потолок масштаба.

---

### 3. Код, зависящий от миграции БД — с fallback'ами с первого коммита

Без fallback = окно ломаного прода между деплоем кода и выполнением миграции владельцем.

**Прецедент**: ededa0f без fallback → окно ~5 мин: топапы начислялись 500 кр вместо правильных, активации плана проходили без кредитов. Hotfix 41cc1de потребовался немедленно.

---

### 4. CC имеет Railway GraphQL (deployments, logs, variables, redeploy)

И `VERCEL_TOKEN` в Railway env. Самопроверка вместо похода владельца в дашборд. «Дашборд Railway недоступен / не могу проверить» — не отговорка.

---

## КОММУНИКАЦИЯ И ИНСТРУКЦИИ

### 5. Всё для ручных действий владельца — готовым текстом в чат

SQL — блоками с подставленными значениями (UUID явно, не `<ваш_uuid>`). Консольные команды — copy-paste ready. Значения берутся из проекта/БД, не запрашиваются у владельца дополнительно.

«Открой файл / см. выше / см. сообщение CC / зайди в дашборд» = невыполненная работа. Архитектор вставляет готовый блок в СВОЮ инструкцию.

**Прецедент**: нарушено ~8 раз за июль, каждый раз ловил владелец.

---

### 6. Пропуск пункта задания без явного «пропустил, потому что...» — хуже невыполнения

Молчаливый пропуск скрывает пробел и блокирует обратную связь.

**Прецеденты**: п.0 разведки админ-панели (молчаливо выброшен → не была найдена прямая запись плана в обход модели); п.2 refund мёртвого прогона (деньги пользователей).

---

### 7. CC не расширяет скоуп молча

Докладывать строкой «расширяю, потому что X» до изменения. Постфактум «владелец всё равно согласился» не считается.

**Прецеденты**: рестарт video-server без проверки активных рендеров; добавление async-движков озвучки без согласования (оказалось санкционировано — но порядок нарушен).

---

### 8. Секреты — только в env, не в чат

Ни архитектору, ни в CC. Даже тестовые ключи ЮKassa, даже «для диагностики». Ключи идут только в env Vercel/Railway.

---

## ДИАГНОСТИКА И ВЕРИФИКАЦИЯ

### 9. Вердикты о состоянии БД и внешних систем — только по первоисточнику

Порядок: сначала факт из БД/лога/curl, потом правка. CC не рассуждает «скорее всего» о состоянии живых данных.

**Прецеденты за 21–22.07 (один баг субтитров, три диагноза подряд)**:
- «Редкий молчаливый сбой createSignedUrl» — реально: вызов ДО загрузки файла.
- «Колонки `type` нет, миграция 005 не выполнена» — колонка была в БД.
- «Причина незачисления неизвестна, нужен ещё платёж» — причина лежала в присланном логе.

---

### 10. Лимиты и блоки внешних систем — только с первоисточником (доки / живой ретест)

Не принимать вердикт CC о возможностях/ограничениях внешних систем без проверки.

**Прецеденты**:
- «Haiku max 8192 output tokens» (реально 64K по документации) — добавление капа 8192 ввело бы реальное ограничение, которого нет.
- «WAF блокирует Railway → R2 навсегда» (реально — TLS-болячка per-account cert, прошла) — Worker-прокси за $5/мес был бы лишним.

---

### 11. Опровергнутые гипотезы не тащить дальше

Факт, указанный владельцем, обновляет вывод в том же ответе.

**Прецедент**: CC повторил старый неверный вывод в ответе, где ему уже дали опровергающий факт (лог с явной причиной).

---

### 12. На lefiro.co фетч CC ненадёжен

Кэш CF возвращает устаревшую копию сайта. Истина о текущем состоянии прода — скрин владельца или его прогон.

**Прецедент**: дважды «изменения не доехали до прода» при рабочем деплое.

---

### 13. Симуляции = прогнанные факты на живой БД

Не написанные ожидания.

Стандарт SQL-симуляции:
1. **BASELINE**: снять текущие значения из БД.
2. **SIM**: выполнить одну операцию с одним ожиданием.
3. **VERIFY**: SELECT подтверждает ожидание.
4. **RESTORE**: `DELETE` / `UPDATE` с **узким WHERE** по тестовым значениям + обоснование почему именно этот WHERE безопасен.

## ПРОДУКТ И UX

### 14. Данные прошлой сессии ≠ выбор юзера

Всё, что запускает платную/тяжёлую работу, включается только явным действием в текущей сессии. Данные из БД прошлой сессии = видимый **выключенный** пресет с подписью «найдены данные прошлой сессии (N блоков) — применить?».

**Прецедент**: фаза-зомби субтитров — старые blocks автоматически включали burn-in → 16-минутный рендер пользователя вместо ожидаемого 5-минутного.

---

### 15. UI-скрины ≠ приёмка функционала

Приёмка = прогон с результатом + перезаход (`?run=` восстанавливает данные, кнопки работают, данные непусты после сохранения).

**Прецедент**: волна 1 инструментов «принята по красивым карточкам» — функционал сохранения, «Использовать в студии», паритет настроек был мёртв.

---

### 16. «Появилось в дашборде» ≠ «данные внутри верны»

Баг записи и баг чтения — разные баги, проверять оба.

**Прецедент**: субтитры показывали «Сохранено ✓» — страница читала `json.data.script` вместо `json.data.project.script`, данные были пустыми. Три раунда debugging.

---

### 17. tool_run — гость в студийном пайплайне

Любой переиспользуемый студийный путь (озвучка, обложка, скрипт) навязывает гостю свой контекст (следующий шаг, статус, `project_id`-обязательность). При переиспользовании — проверять `type` перед тем, как пайплайн достраивает следующий шаг.

**Прецедент**: tool_run-озвучка получала `status='generating_subtitles'` (следующий шаг видео-пайплайна).

---

## РАЗНОЕ

### 18. Среда имеет диалект

`NEXT_PUBLIC_*` — только Vercel. Кросс-копирование паттернов между Vercel-кодом и Railway-кодом = краш.

**Прецедент**: `const APP_URL` из email-блока (скопирован `NEXT_PUBLIC_APP_URL` паттерн в Railway-файл) → `SyntaxError: Identifier 'APP_URL' has already been declared` при загрузке модуля → crash-loop → 4ч24м даун прода.

---

### 19. Кэш CF — только GET'ом

`curl -r 0-1023 -D -`. HEAD всегда возвращает `CF-Cache-Status: DYNAMIC` — это не провал правила, а особенность CF.

---

### 20. user_id берётся из БД/проекта, не из почты

У владельца несколько аккаунтов. Идентификатор тянуть из проекта/платежа/логов, не подставлять email.

**Прецедент**: CC дважды подставил несуществующий `khudyshkin.d@gmail.com` в SQL-запросы начисления кредитов.

---

### 21. Тестовые эндпоинты на проде удаляются сразу после приёмки

Не «потом», не «на всякий случай оставим». Сразу.

Прецедент-стандарт: `subtitles-e2e`, `incident-test` — удалены сразу после приёмки.
